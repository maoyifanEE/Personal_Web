# Personal_Web 项目说明文档

## 0. 文档用途

本文档是 `Personal_Web` 项目的长期项目说明文档，用于统一项目方向、技术路线、架构原则、权限原则、安全原则、开发流程、部署原则和历史记录方式。

本项目主要由 AI / Codex 协助开发，用户负责提出需求、测试功能、反馈问题和验收结果。因此，本文档的核心作用是：

1. 防止项目开发过程中失控；
2. 防止 AI / Codex 一次性大改；
3. 保证代码结构长期清晰；
4. 保证后续功能可以持续扩展；
5. 保证隐私、安全、备份和权限隔离不被忽略；
6. 让每一次开发都有明确目标、测试方式和历史记录。

本文档应长期保留，并随着项目发展少量更新。

## 2. 项目定位

`Personal_Web` 是一个长期使用、持续迭代的私人 Web App 项目。

它不是一次性完成的产品，而是一个会不断增加子应用、不断完善功能的个人工具平台。

项目最终目标是成为一个统一入口，用来管理个人常用工具，例如：

* 备忘录；
* 饮食记录；
* 运动记录；
* 学习记录；
* 打卡工具；
* 个人数据记录；
* 其他后续逐步增加的私人工具。

本项目不追求一次性完成所有功能，而是通过小步迭代逐渐完善。

---

## 3. 核心目标

本项目长期目标如下：

1. 构建一个长期稳定运行的网站；
2. 手机和电脑都能访问；
3. 少数指定用户使用，不开放公众注册；
4. UI 简洁、美观、好用；
5. 架构清晰，方便后续不断添加新功能；
6. 数据存储在云服务器数据库中，实现手机和电脑同步；
7. 重视隐私、安全、备份和权限隔离；
8. 每次开发只做小范围改动，避免一次性大改导致项目失控；
9. 每个功能都应可以独立测试；
10. 项目文档应随着项目发展持续更新。

## 6. 总体架构原则

本项目采用：

> 一个主网站，多个子应用。

不要为每个小功能单独创建独立网站。

所有子应用都应挂在 `Personal_Web` 主网站下，通过路由区分。

长期推荐路由结构：

```text
/
├── /login
├── /hub
│   ├── /hub/apps
│   ├── /hub/settings
│   └── /hub/admin
├── /about
├── /works
└── /articles
```

说明：

* `/`：公开首页；
* `/about`、`/works`、`/articles`：未来公开路由；
* `/login`：Auth Route，登录页面可以未登录访问，但不能暴露私人数据；
* `/hub`：Private Route，未来登录后的主控制台；
* `/hub/settings`：Private Route，未来设置页面；
* `/hub/admin`：Private Route，未来 Owner 管理页面；
* `/hub/apps`：Private Route，未来不同子应用入口。

子应用示例：

```text
/apps/memos
/apps/habits
/apps/food
/apps/exercise
/apps/study
```

不要提前写死所有子应用。只在用户明确提出需求后逐步增加。

## 11. 用户和权限原则

本项目不开放公众注册。

用户由 Owner 手动创建。

至少支持两种角色：

* Visitor；
* Owner；
* Member。

角色命名统一为：

* Visitor：未登录访客，只能访问公开页面，不能访问私人工具；
* Member：未来由 Owner 创建或批准的授权用户，登录后只能访问被授权的私人工具；
* Owner：网站所有者，未来拥有完整管理能力。

权限要求：

1. Owner 可以管理用户；
2. Member 只能访问自己的数据；
3. 默认所有私人数据都必须按 `user_id` 隔离；
4. Member 不能看到其他 Member 的数据；
5. 所有管理页面必须做权限判断；
6. 后续可以扩展更细权限，但早期不要复杂化；
7. Owner 是否能查看 Member 的私人内容，需要后续明确设计，不应默认开放；
8. 所有涉及用户数据的查询都必须检查当前用户身份。

---

## 12. 数据隔离原则

所有私人数据表默认必须包含：

```text
user_id
```

任何查询、修改、删除私人数据的操作，都必须基于当前登录用户进行过滤。

例如：

* A 用户不能看到 B 用户的备忘录；
* A 用户不能修改 B 用户的记录；
* A 用户不能删除 B 用户的数据；
* Member 不能访问 Owner 管理页面；
* 未登录用户不能访问私人页面。

这条规则优先级高于功能便利性。

## 13. 安全原则

必须遵守以下安全原则：

1. 密码不能明文存储，必须哈希；
2. `.env` 文件不能提交到 GitHub；
3. GitHub 仓库必须是 private；
4. 数据库端口不要暴露公网；
5. 全站正式使用时必须使用 HTTPS；
6. 服务器只开放必要端口；
7. 不要在代码里硬编码密钥、密码、token；
8. 上传文件功能后续再做，早期不要贸然开放；
9. 重要操作后续建议记录 audit log；
10. 备忘录等私人内容默认只允许本人访问；
11. 不要把真实密码、token、cookie 发给 AI；
12. `.env.example` 只能写变量名和示例格式，不能写真实密钥。

当前静态阶段不实现真实登录、认证、授权、cookie、token、后端 API 或私人数据存储。隐藏入口只是视觉设计元素，不是安全机制。未来安全必须依赖真实认证、授权、服务端检查和路由保护。

## 14. 数据和备份原则

后续引入数据库后，必须遵守以下原则：

1. 数据库存储在云服务器 PostgreSQL；
2. 所有用户数据必须有 `user_id`；
3. 每次重要数据库变更前必须备份；
4. 需要提供备份脚本；
5. 需要提供恢复说明；
6. 后续应支持定时备份；
7. 不要把重要数据只存在浏览器 localStorage；
8. localStorage 只能用于主题、临时状态等非关键数据；
9. 数据库迁移必须可解释、可测试、可回滚；
10. 生产环境数据库和本地测试数据库应区分。

## 15. UI / UX 原则

长期 UI 原则：

1. 手机优先；
2. 电脑端自适应；
3. 页面简洁，不堆太多信息；
4. 常用操作要少点击；
5. 按钮、输入框、卡片风格统一；
6. 首页应该像控制台，让用户快速进入各子应用；
7. 新功能必须考虑手机端显示；
8. 不做只有电脑端好用的页面；
9. 不使用过度花哨但难维护的动画；
10. UI 要长期可维护，不追求一次性炫技；
11. 错误提示要清楚、可理解；
12. 空状态页面要友好，例如“暂无记录”；
13. 删除、覆盖、批量操作等危险操作要有确认。

---

## 16. PWA 原则

网站后续可以逐步支持 PWA。

PWA 目标：

1. 手机端可以添加到主屏幕；
2. 打开体验接近 App；
3. 常用页面加载快速；
4. 后续可考虑离线缓存；
5. 后续可考虑通知提醒。

注意：

1. 不要一开始为了 PWA 引入复杂系统；
2. 推送通知后续再做；
3. 离线编辑后续再做；
4. 任何涉及数据同步的 PWA 功能都必须谨慎设计，避免数据冲突。

## 17. 部署原则

长期部署目标：

1. 源码放在 GitHub Private Repo；
2. 网站运行在阿里云大陆服务器；
3. 用户通过域名访问网站；
4. 数据库运行在服务器上；
5. 本地电脑不需要一直开机；
6. 每次上线前必须备份数据库；
7. 每次部署后必须确认网站可访问、登录正常、核心功能正常；
8. 服务器只开放必要端口；
9. 正式环境使用 HTTPS；
10. 部署文档必须保存在 `docs/DEPLOYMENT.md`。

每次涉及部署变更时，必须说明：

1. 影响范围；
2. 是否需要停机；
3. 是否需要数据库迁移；
4. 如何验证部署成功；
5. 如何回滚。

6. 把数据库暴露公网；
7. 为了炫技引入复杂架构；
8. 未经确认修改数据库结构；
9. 把临时测试代码留在正式代码中；
10. 假设用户懂复杂运维；
11. 未说明风险就修改部署配置；
12. 未备份就做重要数据库变更；
13. 未检查数据隔离就上线私人数据功能；
14. 未经明确需求提前实现复杂功能。

---

## 25. 项目长期原则总结

`Personal_Web` 的长期原则可以总结为：

> 小步迭代，结构清晰，权限优先，数据安全，长期可维护。

每次开发都应优先保证：

1. 不破坏已有功能；
2. 不泄露用户数据；
3. 不引入无必要复杂度；
4. 不跳过测试；
5. 不让项目方向失控。

本项目宁可慢一点，也要保持长期稳定、清晰、可扩展。

## 静态资源与本地 / 远端一致性原则

`Personal_Web` 项目需要保证本地开发环境和未来远端服务器环境尽量一致。

本项目的基本流程是：

```text
本地编写代码
→ 提交到 GitHub 私有仓库
→ 服务器拉取 GitHub 仓库代码
→ 服务器运行或托管这些文件
→ 用户通过域名访问网站
```

因此，网页运行所需的静态资源原则上都应保存在当前项目文件夹内，并纳入 Git 版本管理。

这样可以保证：

1. 本地打开网页时使用的是项目内资源；
2. 推送到 GitHub 后，资源不会丢失；
3. 服务器拉取代码后，可以得到和本地一致的文件结构；
4. 未来远端网站显示效果和本地测试效果尽量一致；
5. 避免因为引用本地电脑路径或外部资源导致服务器上无法正常显示。

---

### 当前静态网站阶段的资源原则

在当前纯静态网站阶段，网页应尽量使用项目内资源。

推荐结构：

```text
Personal_Web/
├── index.html
├── styles.css
├── script.js
├── assets/
│   ├── icon.svg
│   ├── images/
│   └── fonts/
├── docs/
│   ├── PROJECT_GUIDE.md
│   └── PROJECT_HISTORY.md
└── README.md
```

其中：

* `index.html` 是首页；
* `styles.css` 是本地样式文件；
* `script.js` 是本地脚本文件；
* `assets/` 存放图标、图片、字体等静态资源；
* `docs/` 存放项目文档。

---

### 路径引用原则

网页中引用 CSS、JS、图片、图标等资源时，应使用相对路径。

推荐写法：

```html
<link rel="stylesheet" href="./styles.css">
<script src="./script.js" defer></script>
<link rel="icon" href="./assets/icon.svg">
<img src="./assets/images/example.png" alt="example">
```

不要使用本地电脑绝对路径。

禁止写法：

```html
<link rel="stylesheet" href="C:/Users/xxx/Desktop/styles.css">
<img src="D:/images/example.png">
<script src="C:/Users/xxx/script.js"></script>
```

原因是：

```text
C:/Users/xxx/...
D:/...
```

这类路径只在用户自己的电脑上存在。代码推送到 GitHub 或部署到服务器后，服务器上没有这些路径，网页资源会加载失败。

---

### 外部资源使用原则

当前阶段尽量不要使用外部资源，例如：

```html
<script src="https://example.com/xxx.js"></script>
<link rel="stylesheet" href="https://example.com/xxx.css">
```

原因：

1. 外部资源可能失效；
2. 外部资源可能加载慢；
3. 外部资源可能被墙或被限制；
4. 外部资源会让本地和远端表现不一致；
5. 外部资源会增加隐私和安全风险。

如果未来确实需要使用 CDN、第三方字体、地图、统计、图标库等外部服务，必须单独说明原因，并经过确认后再引入。

---

### GitHub 中应该保存的内容

以下内容应该放进项目文件夹，并提交到 GitHub：

```text
HTML 文件
CSS 文件
JavaScript 文件
本地图标
本地图片
本地字体
项目文档
部署脚本
备份脚本
README
.gitignore
```

例如：

```text
index.html
styles.css
script.js
assets/icon.svg
assets/images/xxx.png
docs/PROJECT_GUIDE.md
docs/PROJECT_HISTORY.md
README.md
```

---

### GitHub 中不应该保存的内容

以下内容不要提交到 GitHub：

```text
真实 .env 文件
服务器密码
数据库密码
API token
node_modules
数据库实际数据文件
系统缓存文件
用户上传的大文件
本机生成的桌面快捷方式 .lnk
```

如果需要提供环境变量格式，只能提交：

```text
.env.example
```

不能提交：

```text
.env
```

`.env.example` 只能写变量名和示例格式，不能写真实密码或真实密钥。

---

### 桌面快捷方式原则

如果本地需要桌面快捷方式，推荐提交“创建快捷方式的脚本”，而不是提交本机生成出来的 `.lnk` 文件。

推荐提交：

```text
scripts/create_desktop_shortcut.ps1
```

不推荐提交：

```text
Personal_Web.lnk
```

原因是 `.lnk` 往往包含当前电脑上的绝对路径，例如：

```text
C:\Users\某个用户\Desktop\Personal_Web\index.html
```

这个路径换一台电脑或部署到服务器后就无效。

因此：

1. `.ps1` 创建脚本可以提交；
2. `.lnk` 快捷方式文件不建议提交；
3. 如果项目文件夹移动位置，应重新运行脚本创建快捷方式。

---

### 未来使用 Next.js 阶段的资源原则

如果未来项目从纯静态网站升级到 Next.js，则静态资源应优先放在：

```text
public/
```

例如：

```text
public/
├── favicon.ico
├── assets/
│   ├── icon.svg
│   ├── images/
│   └── fonts/
```

Next.js 页面中可以通过如下方式引用：

```text
/assets/icon.svg
/assets/images/example.png
```

但无论当前是纯静态网站，还是未来使用 Next.js，都应遵守同一个原则：

> 网页运行所需的资源应保存在项目文件夹中，并随代码一起进入 GitHub 和服务器，除非该资源明确设计为外部服务。

---

### 核心原则总结

`Personal_Web` 的静态资源原则是：

> 本地能打开，GitHub 有资源，服务器拉取后也能打开。

具体要求：

1. 不依赖本机绝对路径；
2. 不随意依赖外部 CDN；
3. 网页资源统一放在项目文件夹；
4. 使用相对路径引用本地资源；
5. 重要资源必须提交到 GitHub；
6. 密钥、密码、数据库数据不能提交到 GitHub；
7. 本地生成的快捷方式不作为项目资源提交；
8. 保证本地文件结构和未来服务器文件结构尽量一致。
